# 统一站点时钟与事件流设计

日期：2026-06-15

适用项目：智能充电桩调度计费系统

目标版本：站点运行控制台

## 1. 背景

上一版系统已经有车主端、管理员端、站点运行台、课程样例播放和课程核对能力。但现在暴露出一个更根本的问题：系统看起来有时间轴、有事件进度、有播放控制，实际业务却仍然像几组互不相干的页面。

具体问题如下：

1. 车主提交申请后，如果没有管理员手动点击调度，车辆会停在等待区，系统没有形成自然的运行推进。
2. 课程样例被放在单独的推演入口里，显得像验收专用工具，不像正常业务能力。
3. 事件进度更像快照列表，而不是真正的站点时间推进。
4. 实时数据、课程数据、未来订单和人工测试事件之间没有统一入口。
5. 当前的“实时”和“推演”容易被理解成两套系统，而不是同一个站点在不同时间策略下运行。

这个设计的核心修正是：系统不再把“实时”和“推演”当作割裂模式，而是维护一个统一的站点时钟和一条统一的站点事件流。无论事件来自车主操作、管理员操作、导入文件、课程样例、外部计划订单，还是手动生成，都进入同一套运行内核。页面看到的站点状态，是这个内核在当前站点时间上的投影结果。

本文在概念上取代 `2026-06-14-unified-station-time-system-design.md` 中“实时 / 推演”作为顶层模式的表达。旧文档中的时间控制、可视化库取舍和性能约束仍可参考，但实现时应以本文的“统一站点时钟 + 统一事件流”为准。

## 2. 设计目标

本次设计要解决的是系统边界，而不只是文案或界面装饰。

目标如下：

1. 建立统一的 `StationClock`，让站点时间可以正常流动、暂停、倍速、跳转和手动指定。
2. 建立统一的 `StationEvent`，让所有来源的订单、故障、恢复、调度和完成动作进入同一事件流。
3. 让课程样例只是“导入事件序列”的一种预设来源，而不是独立验收模式。
4. 让正常使用也能从当前时间生成事件、导入未来订单、同步计划订单。
5. 让运行推进真实发生：车辆开始充电后，电量按站点时间增长；充满后离开；队列车辆进入空闲充电桩。
6. 让回放、预测、课程复盘成为站点运行控制台上的功能，而不是割裂页面。
7. 为后续更精致的时间轴、事件回放和性能优化留下清晰接口。

## 3. 非目标

第一期不做以下内容：

1. 不接入真实充电桩硬件。
2. 不实现完整生产级事件溯源系统。
3. 不把每一秒状态都持久化到数据库。
4. 不直接引入 Open MCT、Cesium 或大型三维场景框架。
5. 不实现复杂预测模型，只先支持人工导入或预置的未来事件。
6. 不让临时推演结果自动覆盖真实业务数据。

## 4. 核心原则

### 4.1 只有一个站点运行内核

系统不应有“实时一套逻辑、课程一套逻辑、预测一套逻辑”。所有场景都应经过同一个运行内核：

```text
事件来源 -> 标准站点事件 -> 站点时钟推进 -> 调度计费规则 -> 站点状态投影
```

差异只体现在事件来源、事件状态和时钟策略上。

### 4.2 站点时间不等于系统真实时间

“实时”不应被理解成只能跟随电脑系统时间。对本项目来说，更准确的概念是“站点时间”。站点时间可以采用不同策略：

| 策略 | 含义 | 典型用途 |
|---|---|---|
| 跟随当前时间 | 站点时间按真实秒推进，倍率为 1x | 普通课堂演示、正常业务操作 |
| 暂停 | 站点时间停在某个时刻 | 讲解当前状态、查看某个事件 |
| 倍速 | 站点时间按 5x、10x、60x 推进 | 快速看完充电过程 |
| 指定时间 | 手动把站点时间设到某个时刻 | 从课程 06:00 开始、跳到 08:30 |
| 事件跳转 | 跳到上一条或下一条事件发生时刻 | 回放、排查、核对 |

因此页面不需要把“实时”和“推演”做成两个互斥世界。用户只是在同一个站点运行控制台里选择当前站点时钟的策略。

### 4.3 导入数据不是特殊验收入口

老师给的样例、同学手工整理的测试数据、外部系统同步的预约订单，本质上都是事件来源。课程样例可以作为一个预设按钮出现，但主语应该是“导入事件序列”或“同步计划订单”，而不是“加载课程”。

推荐文案：

```text
导入事件序列
同步计划订单
添加当前请求
添加未来请求
课程样例：第二次作业验收用例
```

不推荐文案：

```text
加载课程
验收模式
演示数据
执行一次调度
```

## 5. 概念模型

### 5.1 StationClock

`StationClock` 是站点运行的唯一时间来源。

核心字段：

```text
currentTime
baseWallTime
baseStationTime
rate
running
timePolicy
windowStart
windowEnd
lastAdvancedAt
```

字段说明：

| 字段 | 含义 |
|---|---|
| `currentTime` | 当前站点时间 |
| `baseWallTime` | 上一次校准时的真实系统时间 |
| `baseStationTime` | 上一次校准时的站点时间 |
| `rate` | 时间倍率，1 表示正常秒流动，10 表示十倍速 |
| `running` | 时钟是否正在推进 |
| `timePolicy` | `FOLLOW_WALL_TIME`、`MANUAL`、`PLAYBACK` 等 |
| `windowStart` / `windowEnd` | 当前关注的时间窗口 |
| `lastAdvancedAt` | 后端上一次推进运行内核的真实时间 |

核心公式：

```text
running = true 时：
currentTime = baseStationTime + (now - baseWallTime) * rate

running = false 时：
currentTime = baseStationTime
```

### 5.2 StationEvent

`StationEvent` 是所有输入的统一表达。

核心字段：

```text
id
eventTime
receivedAt
sourceType
sourceName
eventType
payload
commitState
sequence
actorType
actorId
originId
```

字段说明：

| 字段 | 含义 |
|---|---|
| `eventTime` | 事件在站点时间线上的发生时间 |
| `receivedAt` | 系统接收到事件的真实时间 |
| `sourceType` | 事件来源类型 |
| `sourceName` | 对用户可见的来源名称 |
| `eventType` | 事件类型 |
| `payload` | 事件载荷 |
| `commitState` | 事件是否已写入当前站点运行 |
| `sequence` | 同一时间内的稳定排序 |
| `originId` | 导入批次、外部订单、课程样例或业务记录来源 |

推荐来源类型：

```text
MANUAL_OPERATION
PRESET_OPERATION
IMPORTED_SEQUENCE
SCHEDULED_ORDER_SYNC
COURSE_PRESET
SYSTEM_DERIVED
```

推荐事件类型：

```text
VehicleRegistered
ChargeRequestSubmitted
ChargeModeChanged
RequestedAmountChanged
VehicleQueued
DispatchRequested
VehicleAssignedToPile
ChargingStarted
ChargingCompleted
PileFaulted
PileRecovered
BillGenerated
```

`COURSE_PRESET` 只是 `IMPORTED_SEQUENCE` 的一种预置来源。业务规则不应该因为来源是课程样例就走另一套逻辑。

### 5.3 StationProjection

`StationProjection` 是运行内核在某个站点时间上的输出。

```text
StationClock + StationEvent[] + 调度计费规则 -> StationSnapshot
```

`StationSnapshot` 应包含：

```text
snapshotTime
clockState
sourceSummary
waitingArea
fastPiles
tricklePiles
vehicles
activeCharges
bills
events
metrics
runtimeNotes
```

页面只渲染 `StationSnapshot`，不直接重复调度计费规则。

## 6. 运行推进机制

这是本次设计最关键的部分。时间控制不能只是播放几个快照，而必须真正驱动站点状态变化。

### 6.1 每次推进做什么

后端每次刷新或播放 tick 时，按以下顺序推进：

1. 根据 `StationClock` 计算新的站点时间。
2. 找出 `(lastProjectedTime, currentTime]` 内发生的外部事件。
3. 按 `eventTime + sequence` 顺序应用这些事件。
4. 按经过的站点时间更新正在充电车辆的已充电量。
5. 如果车辆达到目标电量，生成 `ChargingCompleted` 事件。
6. 生成账单或明细，释放充电桩。
7. 对空闲充电桩和等待队列执行调度。
8. 如果有车辆进入充电桩，生成 `VehicleAssignedToPile` 和 `ChargingStarted` 事件。
9. 输出新的 `StationSnapshot`。

伪流程：

```text
advanceRuntime(fromTime, toTime):
  applyExternalEvents(fromTime, toTime)
  updateChargingProgress(fromTime, toTime)
  completeFinishedVehicles()
  freeCompletedPiles()
  dispatchWaitingVehicles()
  startNewChargingSessions()
  generateSnapshot(toTime)
```

### 6.2 充电进度如何可见

如果车辆在 06:10 开始充电，目标 30 kWh，快充功率 30 kW，那么：

```text
每推进 1 个站点小时，车辆增加 30 kWh
每推进 1 个站点分钟，车辆增加 0.5 kWh
10x 倍速时，真实 6 秒约等于站点 1 分钟
```

前端显示应随 `StationSnapshot` 或轻量 tick 更新：

```text
车辆 CAR-1：12.5 / 30.0 kWh
预计完成：06:58
剩余：18 分钟
```

当当前站点时间到达完成时刻后，车辆自动离开，下一辆等待车辆进入充电桩。老师或同学应该能肉眼看到“时间推进 -> 电量增长 -> 完成离场 -> 队列前进”的连续过程。

### 6.3 外部事件和系统派生事件

用户导入或手动添加的是外部事件；系统根据规则生成的是派生事件。

例如导入一条请求：

```text
06:00 ChargeRequestSubmitted(CAR-1, FAST, 30kWh)
```

系统可能派生：

```text
06:00 VehicleQueued(CAR-1)
06:00 VehicleAssignedToPile(CAR-1, F1)
06:00 ChargingStarted(CAR-1, F1)
07:00 ChargingCompleted(CAR-1, F1)
07:00 BillGenerated(CAR-1)
```

派生事件不需要用户手工写入，但应在事件流中可见，这样系统不是黑箱。

## 7. 页面信息架构

### 7.1 顶栏

顶栏不应写技术栈或“验收演示版”。推荐：

```text
波普特大学充电站
调度计费运行控制台
```

右侧显示系统状态：

```text
站点时间 06:25:14
运行中 · 10x
H2 本机数据
```

技术栈可以放到关于、页脚或文档里，不应占据主视觉。

### 7.2 主工作区

推荐布局：

```text
顶部：站点时钟控制条
左侧：事件来源与快捷操作
中间：站点地图和车辆队列
右侧：事件流与运行检查
底部：时间轴
```

其中站点时钟控制条是核心，不是装饰：

```text
[开始/暂停] [1x/5x/10x/60x] [指定时间] [上一事件] [下一事件] [重置到起点]
当前站点时间：2026-06-15 06:25:14
事件窗口：06:00 - 09:30
```

### 7.3 事件来源面板

事件来源面板负责把所有输入统一起来。

建议操作：

```text
添加当前请求
添加未来请求
导入事件序列
同步计划订单
清空当前运行数据
```

`导入事件序列` 里可以提供预设：

```text
课程样例：第二次作业验收用例
空白序列
自定义 Excel / CSV
```

这样老师给的文件仍然很好用，但它不再是一个特殊世界。

### 7.4 站点地图

站点地图展示当前投影状态：

```text
等候区
快充桩 F1/F2
慢充桩 T1/T2/T3
车辆卡片
充电进度
预计完成时间
队列顺序
```

车辆卡片应能区分来源：

| 来源 | 展示 |
|---|---|
| 手动操作 | 普通卡片 |
| 预设操作 | 普通卡片，详情里显示来源 |
| 导入序列 | 卡片详情显示导入批次 |
| 课程样例 | 卡片详情显示“课程样例来源” |
| 系统派生 | 不作为车辆来源，只在事件流展示 |

### 7.5 事件流与运行检查

右侧不叫“验收”，建议叫：

```text
事件流
运行检查
```

事件流显示：

```text
06:00 CAR-1 提交快充申请 30kWh
06:00 CAR-1 进入 F1 开始充电
06:20 CAR-2 进入等候区
07:00 CAR-1 完成充电，生成账单
07:00 CAR-2 进入 F1 开始充电
```

运行检查显示：

```text
当前时刻状态
当前调度规则说明
课程样例核对结果
异常提示
账单摘要
```

只有当导入来源包含课程样例时，才显示课程核对。标题应是“课程样例核对”，不应是“验收”。

### 7.6 车主端交互必须重做

车主端不能再把“注册用户、设置车辆、提交申请、查看状态、查看账单”拼在同一个表单面板里。这种界面没有真实产品逻辑，也会让老师一眼看出只是为了调用接口临时堆出来。

车主端应按真实任务链路设计：

```text
未登录
  -> 登录 / 注册
  -> 车主首页
  -> 我的车辆
  -> 发起充电申请
  -> 当前排队 / 充电状态
  -> 账单与历史记录
```

登录后的车主首页应回答三个问题：

1. 我的车现在是否有进行中的申请或充电会话。
2. 我是否已经登记车辆，是否可以立即发起申请。
3. 最近账单、排队状态或异常通知是否需要处理。

推荐页面结构：

```text
车主首页
我的车辆
充电申请
当前状态
账单记录
账户设置
```

如果还没有车辆，首页应优先引导“添加车辆”。如果已有车辆但没有申请，首页应突出“发起充电申请”。如果正在排队或充电，首页应突出当前状态，而不是继续显示一张空申请表。

### 7.7 车主状态机

车主端应按状态切换主内容：

| 状态 | 主界面重点 | 主要动作 |
|---|---|---|
| `Anonymous` | 登录、注册 | 登录、创建账户 |
| `OwnerNoVehicle` | 尚未登记车辆 | 添加车辆 |
| `OwnerReady` | 已有车辆，无进行中申请 | 发起充电申请、查看历史 |
| `OwnerWaiting` | 已提交申请，正在等待 | 查看排队号、修改申请、取消申请 |
| `OwnerCharging` | 正在充电 | 查看桩号、进度、预计完成、结束充电 |
| `OwnerCompleted` | 本次充电已完成 | 查看账单、返回首页、再次申请 |

这个状态机比“流程 1/2/3/4”更重要。界面不需要长期展示一个固定流程条；只有在提交申请这种多步任务里，才使用短流程提示。

### 7.8 车主申请流程

发起充电申请应是一个独立任务，而不是和注册、车辆设置混在一起。

推荐步骤：

```text
选择车辆
  -> 选择充电模式
  -> 输入申请电量
  -> 查看预计等待和计费规则
  -> 确认提交
  -> 跳转当前状态
```

如果只有一辆车，可以默认选中，但仍应让用户知道申请绑定的是哪辆车。提交成功后，不应停留在申请表单，而应进入“当前状态”，展示排队号、前车数量、预计等待、当前站点时间和可执行动作。

### 7.9 角色入口和页面外壳

不应把 `站点运行`、`车主端`、`管理员端`简单做成同一个页面顶部的几个选项卡。它们是不同角色的工作台：

| 角色 | 入口 | 页面外壳 |
|---|---|---|
| 游客 / 未登录用户 | 登录页 | 简洁账户入口 |
| 车主 | `/owner` | 车主自助门户 |
| 管理员 | `/admin` | 运营管理工作台 |
| 汇报 / 监控人员 | `/station` | 站点运行控制台 |

课程项目可以用本机模拟登录，不必实现复杂权限系统。但界面结构要像真实系统：不同角色有独立首页、导航和任务重心，而不是在一个全局 tabs 里来回切换。

推荐做法：

1. 首屏提供登录和角色快捷入口。
2. 车主登录后进入车主自助门户。
3. 管理员进入运营管理工作台，重点是队列、桩、调度、故障、账单。
4. 站点运行控制台作为监控和汇报入口，重点是站点时钟、事件流和整体状态。
5. 角色之间可以通过账户菜单切换，但不应作为主导航 tabs 平铺。

### 7.10 参考产品观察

本项目不直接复制商业产品视觉，但应学习它们的信息架构。

公开资料中的共同点：

1. [Electrify America Mobile App](https://www.electrifyamerica.com/mobile-app/) 将查找充电站、实时可用性、启动充电、跟踪进度、通知和充电历史放在账户化 app 体验里。
2. [Electrify America FAQ](https://www.electrifyamerica.com/mobile-faq/) 把充电状态、开始/停止通知、最终收据和账户里的充电历史串成一次会话生命周期。
3. [ChargePoint app](https://play.google.com/store/apps/details?id=com.coulombtech&hl=en_US) 强调查找可用站点、开始/跟踪/支付充电、工作场所等候列表，以及把家庭、工作、公共充电活动放在同一处查看。
4. [Tesla Supercharging Support](https://www.tesla.com/support/charging/supercharging) 把支付方式、充电历史和发票放在 Tesla app 的 Charging 区域里，账单不是散落在申请表单下方。

这些参考说明：车主端应围绕账户、车辆、当前会话和历史账单组织，而不是围绕接口按钮组织。

## 8. 数据流

### 8.1 手动当前请求

```text
用户点击添加当前请求
  -> 使用当前站点时间作为 eventTime
  -> 写入 StationEvent
  -> 推进运行内核
  -> 刷新 StationSnapshot
```

### 8.2 手动未来请求

```text
用户填写未来时间和车辆信息
  -> 写入 eventTime 在未来的 StationEvent
  -> 当前站点时间未到时不进入队列
  -> 时钟推进到 eventTime 后自动生效
```

这使“预测”和“预约计划”自然统一。未来事件不是假数据，只是尚未到达站点时间。

### 8.3 导入课程样例

```text
用户选择导入事件序列
  -> 选择预设：课程样例
  -> 系统把样例行转换为 StationEvent
  -> 用户设置站点时间为样例起点
  -> 点击开始或倍速播放
```

课程样例没有独立运行逻辑。它只是一批事件。

### 8.4 回看历史

第一期可以不完整实现真实历史回看，但架构应允许：

```text
用户选择一个过去时间
  -> 系统按事件流投影该时刻状态
  -> 用户可以继续播放或跳转事件
```

如果当前真实业务事件日志还不完整，可以先只对导入序列支持完整回看。

## 9. API 草案

### 9.1 站点时钟

```text
GET /api/station/clock
PATCH /api/station/clock
POST /api/station/clock/play
POST /api/station/clock/pause
POST /api/station/clock/reset
```

`PATCH /api/station/clock` 示例：

```json
{
  "currentTime": "2026-06-15T06:00:00",
  "rate": 10,
  "running": true,
  "windowStart": "2026-06-15T06:00:00",
  "windowEnd": "2026-06-15T09:30:00"
}
```

### 9.2 事件流

```text
GET /api/station/events
POST /api/station/events
POST /api/station/events/import
POST /api/station/events/preset
DELETE /api/station/events
```

`POST /api/station/events/import` 示例：

```json
{
  "sourceType": "COURSE_PRESET",
  "sourceName": "第二次作业验收用例",
  "resetBeforeImport": true
}
```

`POST /api/station/events` 示例：

```json
{
  "eventTime": "2026-06-15T06:20:00",
  "eventType": "ChargeRequestSubmitted",
  "sourceType": "MANUAL_OPERATION",
  "payload": {
    "carId": "CAR-7",
    "ownerName": "Student",
    "mode": "FAST",
    "requestedKwh": "30.00"
  }
}
```

### 9.3 站点快照

```text
GET /api/station/snapshot
POST /api/station/advance
```

`GET /api/station/snapshot` 返回当前站点时间上的投影。

`POST /api/station/advance` 用于前端播放 tick 或后端定时推进：

```json
{
  "toTime": "2026-06-15T06:30:00"
}
```

如果时钟处于运行中，后端也可以在每次读取 snapshot 前自动推进到当前站点时间。

## 10. 数据存储草案

第一期仍使用 H2/SQLite 风格即可，不需要完整事件溯源库。

新增或调整表：

```text
station_clock
station_event
station_runtime_state
charging_session
bill
import_batch
```

建议：

1. `station_event` 保存外部事件和关键系统派生事件。
2. `charging_session` 保存当前和历史充电过程，便于快速计算进度和账单。
3. `station_runtime_state` 保存当前投影基线，避免每次从零重放。
4. `import_batch` 保存导入来源和说明，用于界面展示和清理。

不要每秒写一条快照。时间推进时只写业务关键点，例如开始充电、完成充电、生成账单、故障、恢复和调度变化。

## 11. 与现有系统的衔接

当前系统已有一些可复用部分：

| 现有部分 | 新位置 |
|---|---|
| 车主端申请 | 生成 `ChargeRequestSubmitted` 事件 |
| 管理员执行调度 | 生成或触发 `DispatchRequested`，随后派生分配事件 |
| 课程样例服务 | 改为事件导入适配器 |
| 课程核对结果 | 放到运行检查中 |
| 站点快照接口 | 改为当前站点时间投影结果 |
| 播放控制组件 | 接入 `StationClock` |

迁移时不需要一次性重写所有代码。可以先让新 API 包装旧服务，再逐步把旧服务内核收敛到事件流。

## 14. 性能设计

性能边界必须从第一期就确定，否则时间系统很容易越做越卡。

原则：

1. 不持久化每秒快照。
2. 后端只在业务关键点写事件和状态。
3. 充电进度按时间差计算，而不是写大量增量记录。
4. 事件流按时间窗口分页或截断展示。
5. 长序列导入时使用批次号，便于一次性清理。
6. 前端播放时只刷新必要组件，不重绘整页。
7. 课程样例这类短序列可以预计算关键点，长序列改为按需推进。

第一期数据量很小，不需要复杂缓存。但 `advanceTo(time)` 的接口边界要保留，以便后续改成检查点和增量投影。

## 15. 风险与处理

| 风险 | 影响 | 处理 |
|---|---|---|
| 抽象过大 | 影响交付速度 | 第一阶段只做站点时钟、事件导入、充电推进和自动队列 |
| 真实业务和导入序列混淆 | 用户不知道数据从哪里来 | 每条事件带来源，界面显示导入批次 |
| 倍速运行破坏数据准确性 | 账单不可信 | 后端按站点时间精确计算，前端动画只做展示 |
| 清空数据误删 | 影响同学演示 | 清空按钮明确作用于当前本机运行数据，可二次确认 |
| 旧课程核对失效 | 影响老师检查 | 保留核对逻辑，只改变入口和文案 |
| 长时间运行卡顿 | 页面体验差 | 不生成每秒快照，事件流分页，状态按需投影 |

## 16. 最小成功标准

实现后应能完成以下演示：

1. 打开系统，看到的是统一的站点运行控制台。
2. 设置站点时间为 06:00。
3. 通过“导入事件序列”选择课程样例。
4. 点击播放并选择 10x 或 60x。
5. 车辆按事件时间出现，进入等待区或充电桩。
6. 正在充电车辆的电量持续增长。
7. 车辆充满后自动离开并生成账单。
8. 等候车辆自动进入空闲充电桩。
9. 用户可以手动添加当前请求或未来请求。
10. 课程核对只作为运行检查的一部分出现。
11. 车主端像一个登录后的自助门户，能清楚看到车辆、当前状态、申请入口和账单记录。
12. 管理员端像一个运营工作台，能清楚处理队列、桩状态、调度和异常。
13. 站点运行控制台像一个监控和汇报入口，不和车主表单、管理员表格缝合在一起。

一句话说明给老师：

```text
我们的系统维护统一的站点时钟和事件流；车主操作、管理员操作、导入订单和课程样例都会转成同一种站点事件，系统按站点时间推进调度和计费，所以可以正常运行，也可以暂停、倍速、跳转和复盘。
```

## 17. 后续实现顺序

建议下一步按以下顺序实施：

1. 后端加 `StationClockService` 和基础 API。
2. 后端加 `StationEventService`，先支持手动事件和课程样例导入。
3. 后端改造运行推进，让充电进度、完成、释放、队列进入自动发生。
4. 前端重做站点运行控制台的信息架构和文案。
5. 前端接入播放、暂停、倍速、指定时间。
6. 前端把课程样例作为导入预设，而不是独立入口。
7. 前端重做车主自助门户和运营管理工作台，拆掉全局 tabs 式角色切换。
8. 补齐测试和浏览器验证。

这个顺序先解决系统最真实的问题：时间推进必须能驱动业务，而不是只播放静态快照。

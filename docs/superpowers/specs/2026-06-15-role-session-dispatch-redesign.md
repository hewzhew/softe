# 多角色工作台与故障感知调度重设计

日期：2026-06-15

适用项目：智能充电桩调度计费系统

目标版本：课程验收前的完整本机演示系统

## 1. 背景

当前系统已经具备站点时钟、事件流、车主申请、调度、故障、计费和课程样例导入能力。但随着验收要求变细，现有结构暴露出几个根本问题：

1. 前端仍是 `站点运行 / 车主自助 / 运营管理` 三个顶层 tab，角色边界像演示拼接，不像真实系统。
2. 车主登录只是车辆建档后的局部状态，没有统一登录入口、角色权限和可刷新会话。
3. 管理员缺少营收、账单、单点调度和完整运营视角。
4. 当前调度策略能把车放进队列，但对局部队列、故障、恢复、手动调度的解释不够完整。
5. 故障恢复后，如果已排队车辆不能重新利用恢复的充电桩，会出现“新来的车反而能用空桩，早来的车继续排队”的不合理现象。
6. 充电中修改电量需要重新定义成“修改目标电量”，否则会和已经发生的充电事实冲突。

本设计把系统重心从“验收演示页面”调整为“多角色站点运行系统”：同一个后端站点状态由车主端、管理员端和站点沙盘共同观察和操作；课程数据只是事件来源之一；调度策略对正常申请、单点调度、故障和恢复采用同一套可解释规则。

## 2. 设计目标

目标如下：

1. 提供统一登录入口，按账号角色进入车主端或管理员端。
2. 支持多个浏览器 tab 登录不同账号，同时刷新当前 tab 不丢登录态。
3. 重做前端信息架构，取消三角色顶层 tab，改为登录后的角色工作台。
4. 将站点沙盘抽为共享组件，车主端看简化状态，管理员端看完整运营态势。
5. 管理员端增加营收、账单、单点调度、自动补齐调度、故障处理和运行状态面板。
6. 车主端支持查看当前状态、提交申请、修改模式、修改目标电量、结束充电、查看账单。
7. 充电中允许修改目标电量，但必须满足 `已充电量 <= 新目标电量 <= 电池容量`。
8. 升级调度策略，使其同时考虑活动充电剩余时间、桩队列负载、局部队列容量、故障状态和恢复重排。
9. 后端集中校验业务规则，前端只做友好提示，不承担最终规则判断。
10. 保留本机演示可控性，不引入过重的生产级认证框架或运筹优化库。

## 3. 非目标

第一期不做以下内容：

1. 不接入真实充电桩硬件和真实支付。
2. 不实现生产级 OAuth、JWT 刷新令牌、细粒度 RBAC 权限系统。
3. 不实现复杂全局最优运筹模型、强化学习或线性规划求解器。
4. 不把每一秒的站点状态都持久化成快照。
5. 不把车主端和管理员端拆成两个独立前端工程。
6. 不追求生产级高并发，只保证课程验收数据规模下状态正确、界面清楚、规则可解释。

## 4. 外部参考与设计取舍

调度策略参考三类资料，但只吸收适合本项目规模的部分：

1. 在线 EV 充电调度研究通常强调车辆到达的不确定性、站点容量约束和到达时承诺。本系统不做复杂社会福利优化，但采用“每次事件到达后滚动重算”的在线调度思想。
2. 等待时间估计研究通常会把当前排队、服务时间和充电站可用性纳入判断。本系统用“当前充电剩余时间 + 桩队列预计时间”作为每个桩的负载估计。
3. OCPP 等充电桩管理协议会把充电桩状态区分为可用、充电中、故障等。本系统不实现 OCPP，但管理员端和调度策略应明确区分 `RUNNING / FAULTED` 等状态，故障桩不参与调度。

参考资料：

- Online EV Charging Scheduling With On-Arrival Commitment: https://solar.cs.umass.edu/publications.html
- Reducing Waiting Times at Charging Stations with Adaptive Electric Vehicle Route Planning: https://arxiv.org/abs/2102.06503
- Open Charge Alliance OCPP protocol overview: https://openchargealliance.org/protocols/open-charge-point-protocol/
- Improving Uptime Monitoring with OCPP: https://openchargealliance.org/wp-content/uploads/2024/08/improving_uptime_with_ocpp_v1_2.pdf

## 5. 总体方案

推荐方案是单个 Vue 应用加统一登录入口：

```text
登录页
  ├─ 车主账号 -> 车主自助门户
  └─ 管理员账号 -> 运营管理工作台

共享能力
  ├─ 站点时钟
  ├─ 站点事件流
  ├─ 站点沙盘组件
  ├─ 调度服务
  └─ 计费服务
```

不再把车主端、管理员端和站点运行作为同级 tab 暴露给用户。登录后的角色决定可见页面和可执行操作。

后端保持一个站点运行上下文。车主端、管理员端、课程事件导入和站点沙盘都通过同一套服务读写状态：

```text
用户操作 / 管理员操作 / 导入事件
            │
            v
       后端业务命令
            │
            v
  校验 -> 调度 -> 计费 -> 状态投影
            │
            v
      车主端 / 管理员端 / 沙盘
```

## 6. 登录与会话

### 6.1 账号角色

`UserAccount` 增加角色字段：

| 角色 | 含义 | 典型权限 |
|---|---|---|
| OWNER | 车主 | 管理自己的车辆、提交申请、查自己的状态和账单 |
| ADMIN | 管理员 | 查看全站、调度、故障处理、营收账单、参数配置 |

车主账号绑定车辆。第一期可以是一账号一车，后续可扩展为一账号多车。

系统初始化时种子数据至少包括：

| 账号 | 密码 | 角色 | 说明 |
|---|---|---|---|
| admin | 123456 | ADMIN | 管理员演示账号 |
| CAR-1 | 123456 | OWNER | 默认车主演示账号 |

### 6.2 会话存储

前端当前登录态使用 `sessionStorage`，原因是：

1. 刷新当前 tab 不会丢登录。
2. 多个 tab 可以分别登录不同账号。
3. 不会因为同源 `localStorage` 被覆盖而串号。

`localStorage` 只保存辅助信息，例如上次登录账号、是否记住账号，不保存当前活跃身份。

### 6.3 后端接口

新增认证接口：

```text
POST /api/auth/login
GET  /api/auth/me
POST /api/auth/logout
```

登录成功返回：

```json
{
  "token": "session-token",
  "role": "OWNER",
  "accountId": 1,
  "userName": "Alice",
  "carId": "CAR-1"
}
```

第一期可采用轻量服务端 session 表或内存 session。由于 H2 本机演示重启后数据会重置，session 随后端重启失效是可接受的。

### 6.4 权限边界

后端必须校验权限：

1. OWNER 只能查询和操作自己绑定车辆。
2. ADMIN 可以查询全站账单、营收、站点状态和执行调度/故障操作。
3. 未登录请求只能访问登录、注册或必要的公开配置接口。
4. 前端隐藏按钮不是权限控制，后端必须拒绝越权请求。

## 7. 前端信息架构

### 7.1 路由

继续使用当前 hash 路由即可，不必为了本次重构引入 Vue Router。

建议路由：

```text
#/login
#/owner
#/admin
#/station
```

`#/station` 不作为角色切换 tab，而是共享站点态势页面。第一期可以只允许登录后访问，管理员看到完整控制，车主看到只读态势。

### 7.2 登录页

登录页只做真实入口，不再把注册表单和申请表单混在同一屏。

内容：

1. 账号输入。
2. 密码输入。
3. 登录按钮。
4. 记住账号。
5. 车主注册入口。

车主注册完成后返回登录页或自动登录。

### 7.3 车主自助门户

车主端主线应是“我的车辆现在处于什么状态，我能做什么”。

核心区域：

1. 车辆信息：车牌/车辆编号、电池容量、账号状态。
2. 当前申请：状态、排队号、排队位置、目标电量、当前已充电量、充电桩。
3. 操作区：
   - 无活动请求：提交充电申请。
   - 等待区/桩队列：修改快慢充、修改目标电量、取消申请。
   - 充电中：查看实时电量、修改目标电量、立即结束充电。
   - 已完成：查看账单。
4. 账单记录：只显示当前车主自己的历史账单。
5. 站点概览：只读简化版，帮助车主理解队列和充电进度。

### 7.4 管理员工作台

管理员端主线是“站点是否正常运行，是否需要人工干预，收入和账单是否正确”。

建议左侧导航：

1. 站点沙盘。
2. 调度管理。
3. 故障管理。
4. 账单营收。
5. 参数配置。

管理员首页显示：

1. 当前站点时间和运行状态。
2. 等待区、快充区、慢充区。
3. 各充电桩运行/故障状态。
4. 充电中车辆、桩队列车辆、等待区车辆。
5. 快捷操作：调度下一辆、自动补齐调度、设置故障、恢复故障。

## 8. 目标电量规则

系统将用户可修改的数值统一称为“目标电量”。

### 8.1 提交申请

提交申请时：

```text
0 < 目标电量 <= 车辆电池容量
```

后端拒绝超出容量的申请，不自动缩小。

### 8.2 等待区和桩队列修改

车辆处于 `WAITING_AREA` 或 `PILE_QUEUE` 时：

```text
0 < 新目标电量 <= 车辆电池容量
```

修改后应触发同模式局部重排。原因是目标电量改变会影响所在桩的预计负载和后续车辆等待时间。

### 8.3 充电中修改

车辆处于 `CHARGING` 时：

```text
当前已充电量 <= 新目标电量 <= 车辆电池容量
```

如果 `新目标电量 < 当前已充电量`，后端拒绝，并提示用户应执行“立即结束充电”。

如果 `新目标电量 == 当前已充电量`，可以视为立即达到目标。实现上推荐仍提示用户使用“立即结束充电”，避免隐式结束造成困惑。

如果 `新目标电量 > 当前已充电量`，更新当前请求目标。运行时下一次推进时按新目标判断完成时间。

## 9. 调度模型

### 9.1 基本概念

系统中有两级等待：

1. 站点等待区：尚未分配到具体充电桩的车辆。
2. 充电桩局部队列：已分配到某个充电桩，但尚未开始充电的车辆。

充电中车辆不属于局部队列容量。局部队列容量只限制“排在某个桩后面等待”的车辆数量。

### 9.2 调度触发事件

以下事件触发调度或重排：

1. 提交新申请。
2. 修改目标电量。
3. 修改快慢充模式。
4. 充电完成。
5. 立即结束充电。
6. 充电桩故障。
7. 充电桩恢复。
8. 管理员执行单点调度。
9. 管理员执行自动补齐调度。
10. 导入事件推进到相关业务动作。

### 9.3 充电桩负载估计

对每个可用充电桩计算预计负载：

```text
activeRemainingHours = 当前充电车辆剩余目标电量 / 充电桩功率
queuedHours = sum(局部队列中每辆车目标电量 / 充电桩功率)
candidateHours = 候选车辆目标电量 / 充电桩功率

candidateFinishHours = activeRemainingHours + queuedHours + candidateHours
```

如果充电桩故障，则不参与候选。

如果局部队列已满，则不参与候选。

候选车辆应分配给 `candidateFinishHours` 最小的充电桩。

### 9.4 平局规则

多个充电桩预计完成时间相同时，按以下规则：

1. 局部队列更短的桩优先。
2. 若是故障恢复后的稳定重排，原本就在该桩队列的车辆优先留在原桩。
3. 编号更小的桩优先。

这样能减少无意义挪动，同时保证演示结果稳定。

### 9.5 等待车辆排序

自动补齐调度时，等待车辆排序：

1. 故障中断后剩余目标电量未完成的车辆保留原申请时间。
2. 普通车辆按申请时间升序。
3. 同一申请时间按队列序号升序。

用户主动修改快慢充模式时，进入新模式队列，并生成新队列号。原因是用户改变服务类型，相当于重新选择队列。

用户仅修改目标电量时，不改变原申请时间和队列号。

## 10. 单点调度

管理员端提供三类单点操作：

| 操作 | 含义 |
|---|---|
| 调度下一辆 | 从站点等待区选择最早申请的一辆，按策略分配到可用桩 |
| 调度快充下一辆 / 调度慢充下一辆 | 只在指定模式等待区选择下一辆 |
| 调度此车 | 管理员在等待区表格中指定某辆车进行调度 |

单点调度只移动一辆车。自动补齐调度才会循环调度直到没有可用局部队列位置或等待区为空。

接口建议：

```text
POST /api/scheduler/dispatch-one
POST /api/scheduler/dispatch-car
POST /api/scheduler/dispatch-all
```

请求示例：

```json
{
  "mode": "FAST",
  "carId": "V20"
}
```

`mode` 和 `carId` 可选。只传 `mode` 表示该模式下一辆；只传 `carId` 表示指定车辆；都不传表示全站下一辆。

返回应包含是否成功、失败原因、分配到的充电桩、队列位置和策略说明。

## 11. 故障处理与恢复重排

### 11.1 故障发生

充电桩故障时：

1. 故障桩状态改为 `FAULTED`。
2. 该桩不再参与任何调度。
3. 故障桩上正在充电的车辆结束当前充电 session。
4. 已充电量按站点时间和桩功率计算，生成实际账单或阶段详单。
5. 若车辆目标电量未完成，生成剩余目标电量请求，并保留故障影响标记。
6. 故障桩局部队列中未开始充电的车辆全部退回站点等待区。
7. 对同模式未开始充电车辆执行局部重排。

此规则保证不会让车继续排在故障桩后面。

### 11.2 故障恢复

充电桩恢复时：

1. 充电桩状态改为 `RUNNING`。
2. 该桩重新加入同模式候选资源。
3. 收集同模式所有未开始充电车辆，包括站点等待区和各桩局部队列。
4. 保持正在充电车辆不移动。
5. 清空这些未开始车辆的桩分配。
6. 按当前负载估计重新分配到可用桩。

此规则保证恢复后的空闲桩能服务已经在排队的车辆，而不是只服务新来的车辆。

### 11.3 稳定重排

为了避免恢复时队列跳动过大，重排采用稳定策略：

1. 只重排未开始充电车辆。
2. 已经充电的车辆不迁移。
3. 如果两个方案预计完成时间相同，优先保留原桩。
4. 只有当恢复桩能明显减少等待或填补空闲资源时，车辆才迁移过去。

课程演示规模较小，第一期可以每次恢复都重排同模式未开始车辆；稳定 tie-break 已足够避免随机感。

## 12. 营收与账单

管理员端新增账单营收区。

### 12.1 指标

管理员可以查看：

1. 今日营收。
2. 累计营收。
3. 充电费合计。
4. 服务费合计。
5. 完成订单数。
6. 总充电量。
7. 平均单价。
8. 按快充/慢充分组统计。
9. 按充电桩统计。

### 12.2 查询

接口建议：

```text
GET /api/admin/revenue?from=&to=
GET /api/admin/bills?from=&to=&carId=&pileId=&mode=
GET /api/admin/bills/{billId}/details
```

车主账单接口继续只返回自己的账单。

管理员账单接口返回全站账单，支持按日期、车辆、充电桩、模式筛选。

## 13. 站点状态投影

前端不应各自拼状态。后端提供统一站点快照：

```text
GET /api/station/snapshot
```

快照包含：

1. 当前站点时间。
2. 站点运行状态。
3. 等待区车辆。
4. 快充桩和慢充桩。
5. 每个桩的运行状态、当前车辆、局部队列。
6. 每辆车目标电量、已充电量、预计剩余时间、排队位置。
7. 最近事件。

车主端和管理员端使用同一快照，只是展示层过滤权限和字段。

## 14. 性能约束

本项目不应每秒写数据库快照。推荐策略：

1. 站点时间推进只更新时钟状态。
2. 查询快照时根据当前站点时间即时计算已充电量和预计剩余时间。
3. 只有状态发生边界变化时才写库，例如开始充电、完成、故障、恢复、账单生成。
4. 前端轮询频率控制在 1 秒到 2 秒；倍速播放时也不提高到过高频率。
5. 调度重排只处理未开始充电车辆，避免全量重算所有历史记录。

课程验收数据规模较小，以上策略足够稳定，也便于解释。


其中：

1. `DispatchPlanner` 只负责计算调度方案，不直接写库。
2. `PileLoadEstimator` 只负责计算活动车辆剩余时间和局部队列负载。
3. `SchedulerService` 负责事务内应用调度方案。
4. `FaultAwareRebalanceService` 负责故障和恢复时收集候选车辆并触发重排。
5. `StationSnapshotService` 负责统一站点状态投影。

这样能把调度算法从控制器和普通业务流程中拆出来，便于测试。

## 16. 测试设计

必须增加以下测试：

### 16.1 目标电量

1. 提交申请超过电池容量应拒绝。
2. 等待区修改目标电量超过容量应拒绝。
3. 充电中修改目标电量低于已充电量应拒绝。
4. 充电中修改目标电量高于已充电量且不超过容量应成功。
5. 修改目标电量后预计完成时间随之变化。

### 16.2 调度

1. 空桩优先接收车辆。
2. 有活动车辆时，应把活动车辆剩余时间计入负载。
3. 有局部队列时，应把局部队列目标电量计入负载。
4. 局部队列满时，不应再分配到该桩。
5. 相同预计完成时间时，队列短和编号小的桩优先。
6. 单点调度只移动一辆车。
7. 自动补齐调度循环调度直到没有空间或没有等待车辆。

### 16.3 故障

1. 故障桩不参与调度。
2. 故障桩局部队列车辆退回等待区并重新分配。
3. 故障桩正在充电车辆按已充电量结算或生成阶段详单。
4. 故障恢复后，同模式未开始车辆能利用恢复的空闲桩。
5. 故障恢复不迁移正在充电车辆。

### 16.4 权限

1. 未登录不能访问车主和管理员操作接口。
2. 车主不能查询其他车辆账单。
3. 车主不能执行调度和故障操作。
4. 管理员可以查询全站账单和营收。

### 16.5 前端

1. 刷新车主 tab 后仍保持登录。
2. 两个 tab 分别登录车主和管理员时互不串号。
3. 车主端状态随站点快照刷新。
4. 管理员端单点调度后沙盘更新。
5. 充电中修改目标电量的按钮和错误提示符合后端规则。

## 17. 实施顺序

建议分六步实施：

1. 登录与角色：新增认证接口、sessionStorage 登录态、角色路由守卫，去掉顶层三角色 tab。
2. 车主端重做：把注册、登录、申请、状态、账单拆成清晰工作流。
3. 管理员端重做：站点沙盘、调度管理、故障管理、账单营收分区。
4. 调度内核升级：抽出负载估计和调度规划，支持单点调度和自动补齐。
5. 故障感知重排：故障和恢复统一重排未开始车辆。
6. 目标电量升级：充电中修改目标电量，并补齐状态投影和测试。

若时间紧，最小可验收版本优先级：

1. 登录与角色分离。
2. 管理员单点调度。
3. 故障恢复重排。
4. 充电中修改目标电量。
5. 营收账单。

## 18. 验收讲解口径

可以这样解释系统：

1. 系统不是三个演示页面，而是统一登录后的多角色工作台。
2. 车主和管理员看到的是同一个站点状态，只是权限不同。
3. 调度不是简单找空桩，而是比较每个充电桩的预计完成时间。
4. 局部队列会计入负载，因此不会因为一个桩暂时空或队列短就忽略后续等待时间。
5. 故障发生后，故障桩退出调度；未开始车辆重新进入候选池；正在充电车辆按实际已充电量处理。
6. 故障恢复后，恢复桩重新参与调度，已排队但未开始车辆可以被重新分配过去。
7. 充电中改电量本质是改目标电量，不能小于已充电量；低于已充电量应执行结束充电。
8. 管理员可以通过单点调度人工干预，也可以用自动补齐让系统按策略填充队列。

## 19. 决策结论

采用“统一登录 + 多角色工作台 + 共享站点快照 + 故障感知滚动调度”的方案。

第一期不做复杂优化器，而是实现一个能解释、能测试、能演示的调度策略：

```text
事件触发 -> 收集未开始车辆 -> 估计每个桩负载 -> 选择预计完成最早的可用桩 -> 稳定写入队列
```

这个方案能覆盖验收会追问的关键点：角色分离、会话缓存、单点调度、容量约束、故障恢复、充电中改目标、营收账单和站点状态一致性。

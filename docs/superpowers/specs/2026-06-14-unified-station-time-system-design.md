# 统一站点时间系统设计

日期：2026-06-14

适用项目：智能充电桩调度管理系统

目标版本：站点运行台增强版

## 1. 背景与问题

当前系统已经完成了三类能力：

1. 车主自助和运营管理可以执行真实业务操作。
2. 调度沙盘可以加载课程事件序列并播放站点状态变化。
3. 课程核对可以验证老师样例中的关键结果。

上一版“调度沙盘”已经把“测试用例页面”改成了更自然的运行回放界面，但仍然存在一个架构隐患：如果课程序列只是沙盘里的独立数据，而实时操作只存在于车主端和管理员端，预测又另做一套页面，那么系统会逐渐变成三套体验：

```text
实时使用：用户手动提交请求，站点按当前时间运行
课程回放：加载老师样例，站点按模拟时间播放
预测分析：加载未来订单，站点按未来时间推演
```

这三者表面不同，但本质上都是“某些事件进入充电站，系统在某个时间点投影出站点状态”。因此下一步不应继续做一个更孤立的演示组件，而应抽象出统一的站点时间系统。

本文的核心判断是：实时、回放、预测不应是三套业务，而应是同一个站点运行模型在不同事件来源和不同时间会话下的表现。

## 2. 设计目标

本设计要解决的不是“让验收页面更好看”，而是让系统具备统一的时间和事件表达能力。

目标如下：

1. 把实时操作、课程样例、预测订单都统一为站点事件。
2. 把实时观察、历史回放、未来预测都统一为时间会话。
3. 用同一个状态投影模型渲染站点地图、队列、充电桩、车辆和账单。
4. 保留课程验收核对能力，但把它降级为运行结果检查器，而不是独立业务入口。
5. 支持真实场景中的未来订单，例如预约、批量导入、外部系统同步订单。
6. 支持演示场景中的播放、暂停、倍速、跳转，同时不破坏实时业务的真实语义。
7. 为后续引入更精致的时间轴或可视化库留下接口，但第一期不引入过重框架。

## 3. 非目标

第一期不做以下内容：

1. 不接入真实充电桩硬件。
2. 不把系统改造成完整事件溯源架构。
3. 不要求所有历史业务数据都能从事件日志完整重放。
4. 不直接引入 Open MCT 或 Cesium 作为主应用框架。
5. 不做复杂三维场景。
6. 不让预测结果自动写入实时业务数据。
7. 不为了动画效果牺牲课程样例通过率和普通业务稳定性。

## 4. 概念模型

### 4.1 站点事件

`StationEvent` 表示进入站点运行模型的一条事实或外部输入。

核心字段：

```text
id
sourceType
sourceId
eventTime
receivedAt
actorType
actorId
eventType
payload
commitState
confidence
scenarioId
branchId
sequence
```

字段含义：

| 字段 | 含义 |
|---|---|
| `sourceType` | 事件来源，例如 `LIVE_MANUAL`、`LIVE_PRESET`、`COURSE_SEQUENCE`、`FORECAST_MODEL`、`IMPORTED_PLAN` |
| `eventTime` | 事件在业务时间线上的发生时间 |
| `receivedAt` | 系统接收到事件的真实时间 |
| `eventType` | 事件类型，例如提交请求、取消请求、修改电量、桩故障、桩恢复 |
| `commitState` | `COMMITTED` 表示已进入真实站点，`PROVISIONAL` 表示预测或草案 |
| `confidence` | 预测事件可信度，人工事件默认为 `1.0` |
| `scenarioId` | 课程样例或导入序列标识 |
| `branchId` | 预测分支标识 |
| `sequence` | 同一时间内的稳定排序序号 |

关键原则：

1. 课程样例不是特殊业务，只是一种事件来源。
2. 预测订单不是假数据，只是提交状态为 `PROVISIONAL` 的未来事件。
3. 实时业务也可以产生未来事件，例如预约订单或外部同步的计划订单。
4. 同一个 `eventType` 应尽量复用同一套调度和计费规则。

### 4.2 事件来源

`EventSource` 负责把不同输入转换为标准站点事件。

建议来源类型：

```text
LIVE_MANUAL
LIVE_PRESET
COURSE_SEQUENCE
IMPORTED_PLAN
FORECAST_MODEL
SYSTEM_DERIVED
```

来源说明：

| 来源 | 使用场景 | 是否写入实时业务 |
|---|---|---|
| `LIVE_MANUAL` | 车主端和管理员端真实操作 | 写入 |
| `LIVE_PRESET` | 演示时一键生成当前时间的真实操作 | 可写入 |
| `COURSE_SEQUENCE` | 老师样例、课程验收、课堂复盘 | 默认不写入 |
| `IMPORTED_PLAN` | 导入预约、外部系统同步订单 | 由导入策略决定 |
| `FORECAST_MODEL` | 预测模块生成的未来负载 | 默认不写入 |
| `SYSTEM_DERIVED` | 系统推导出的完成、账单、调度变化 | 跟随所属会话 |

`LIVE_PRESET` 很重要。它可以让演示者在实时模式中点一个“生成当前请求”的按钮，系统仍然走真实业务流程，而不是切到孤立的演示页面。

### 4.3 时间会话

`TimeSession` 表示用户当前以什么方式观察或推演站点。

核心字段：

```text
id
mode
sourceFilter
baselineTime
cursorTime
windowStart
windowEnd
clockRate
playing
branchId
createdAt
```

建议模式：

```text
LIVE
REPLAY
FORECAST
```

模式说明：

| 模式 | 时间来源 | 控制能力 | 典型用途 |
|---|---|---|---|
| `LIVE` | 系统真实时间 | 刷新、冻结视图、查看最近历史 | 正常运行、课堂即时演示 |
| `REPLAY` | 会话模拟时间 | 播放、暂停、倍速、跳转、单步 | 课程样例、历史复盘 |
| `FORECAST` | 未来模拟时间 | 播放、暂停、倍速、跳转、比较分支 | 未来订单推演、调度预案 |

实时模式不能把真实站点“倍速运行”。如果用户想暂停或倍速查看，应创建一个回放会话，回放的是已发生事件或课程事件，而不是暂停真实业务。

### 4.4 状态投影

`StationProjectionRuntime` 接收事件来源和时间会话，输出统一的 `StationSnapshot`。

```text
事件集合 + 会话时间 + 运行配置 -> 站点快照
```

`StationSnapshot` 应服务所有页面：

```text
StationSnapshot
  snapshotTime
  sessionMode
  sourceSummary
  waitingArea[]
  fastPiles[]
  tricklePiles[]
  vehicles{}
  activeSessions[]
  bills[]
  stationMetrics
  ruleNotes[]
  provenance[]
```

其中 `provenance` 用于说明快照来自哪些事件来源，例如“实时操作 3 条、课程序列 36 条、预测分支 1 个”。

### 4.5 预测分支

`ForecastBranch` 表示一组不会立即污染真实站点的未来事件。

核心字段：

```text
id
name
baseSnapshotTime
events[]
metrics
createdBy
createdAt
status
```

状态建议：

```text
DRAFT
RUNNING
COMPLETED
PROMOTED
DISCARDED
```

预测结果只能通过明确动作进入真实业务，例如“应用为预约计划”或“导出调度建议”。第一期可以只展示分支，不做应用动作。

### 4.6 方案比较

本次设计讨论过三种方向。

| 方案 | 做法 | 优点 | 问题 |
|---|---|---|---|
| 独立演示组件 | 保留调度沙盘，只继续美化课程回放 | 改动最小，能快速服务验收 | 演示、实时、预测仍然割裂 |
| 直接引入大框架 | 接入 Open MCT、Cesium 或完整时间轴平台 | 视觉和时间控制能力成熟 | 体量过大，偏离课程系统主体 |
| 统一事件时间模型 | 用事件来源、时间会话、状态投影统一实时、回放、预测 | 业务边界最清楚，可逐步落地 | 需要先写清抽象和迁移路线 |

推荐第三种方案。它不要求第一期重写所有后端，也不排斥后续引入可视化库；它先把“这到底是不是同一个系统”这个核心问题解决掉。

## 5. 总体架构

推荐架构如下：

```text
                       用户操作 / 课程序列 / 导入计划 / 预测模型
                                      │
                                      ▼
                              EventSourceAdapter
                                      │
                                      ▼
                               StationEvent[]
                                      │
                  ┌───────────────────┴───────────────────┐
                  │                                       │
             EventLog / Repositories                 Scenario Memory
             实时业务上下文                            课程/预测上下文
                  │                                       │
                  └───────────────────┬───────────────────┘
                                      ▼
                              TimeSessionService
                                      │
                                      ▼
                          StationProjectionRuntime
                                      │
                                      ▼
                              StationSnapshot
                                      │
             ┌────────────────────────┼────────────────────────┐
             ▼                        ▼                        ▼
          站点运行台                 车主自助                 运营管理
       实时/回放/预测             个人状态和账单            桩状态和队列
```

关键原则：

1. 业务事件是统一输入。
2. 时间会话决定如何观察这些事件。
3. 状态投影是统一输出。
4. 页面不直接判断调度规则，只解释后端投影出的结果。
5. 课程核对是投影后的检查器，不是独立运行内核。

## 6. 三种模式的统一关系

### 6.1 实时模式

实时模式面向正常使用。

```text
车主/管理员操作
  -> 业务命令
  -> StationEvent(sourceType=LIVE_MANUAL, commitState=COMMITTED)
  -> 实时业务数据更新
  -> StationSnapshot(sessionMode=LIVE)
  -> 页面刷新
```

实时模式中的时间来自系统时钟。页面可以一秒一秒更新，也可以手动刷新，但不能用倍速推进真实站点。

实时模式可以提供“预制操作”按钮，例如“生成一辆快充车”“生成一辆慢充车”“模拟 F1 故障”。这些按钮本质上仍然是 `LIVE_PRESET` 事件，默认写入真实业务数据，适合课堂即时演示。

### 6.2 回放模式

回放模式面向课程样例和历史复盘。

```text
选择事件来源
  -> 创建 REPLAY TimeSession
  -> 按 cursorTime 投影快照
  -> 播放 / 暂停 / 单步 / 倍速 / 跳转
```

课程序列只是回放模式的一种事件来源。将来如果保留实时事件日志，也可以回放某天真实运行过程。

回放模式可以一次性生成完整快照包，也可以按时间请求快照。第一期为了稳定演示，继续使用完整回放包；架构上把它理解为一次 `REPLAY TimeSession` 的预计算结果，而不是沙盘专属业务。

### 6.3 预测模式

预测模式面向未来负载和调度预案。

```text
当前真实快照
  + 预约订单 / 导入计划 / 预测模型事件
  -> 创建 FORECAST TimeSession
  -> 生成未来快照和指标
  -> 比较等待时间、桩利用率、收入、故障影响
```

预测模式允许事件时间大于当前时间。未来事件不等于课程假数据，它可以来自真实预约、外部系统计划或人工录入。

预测事件在界面上应有明确视觉区分，例如虚线车辆、半透明事件点、`预测` 标签。用户必须能看出哪些状态已经真实发生，哪些只是推演结果。

## 7. 页面设计

### 7.1 顶级入口

建议把当前 `调度沙盘` 逐步改名为 `站点运行`，形成如下入口：

```text
站点运行
车主自助
运营管理
```

`站点运行` 是全站的时间视图，不再只是课程演示页。它内部提供模式切换：

```text
实时
回放
预测
```

第一期如果担心改名影响现有汇报脚本，可以保留 `调度沙盘` 标签，但页面标题和文档应按 `站点运行台` 设计。

### 7.2 时间控制区

顶部应成为系统最重要的视觉焦点，展示“现在正在看哪个时间”。

推荐内容：

```text
波普特大学充电站
站点运行台

[实时 / 回放 / 预测]
当前时间：08:25:00
事件来源：课程事件序列
状态：播放中 / 已冻结 / 预测分支
```

实时模式按钮：

```text
刷新
冻结视图
查看最近 30 分钟
生成快充请求
生成慢充请求
模拟故障
```

回放模式按钮：

```text
加载事件序列
播放
暂停
上一步
下一步
重置
倍率
跳转
```

预测模式按钮：

```text
新建预测
导入计划
播放
暂停
倍率
比较分支
丢弃分支
```

按钮随模式变化，不应把所有按钮同时堆在页面上。

### 7.3 时间轴

时间轴不只显示课程事件，还应能显示多个事件泳道。

建议泳道：

```text
车主请求
调度变化
充电桩状态
账单生成
预测事件
课程核对点
```

课程事件可以显示在“车主请求”和“充电桩状态”泳道中，课程核对点只作为附加标记出现。这样“核对”不会成为主角。

### 7.4 站点地图

站点地图继续作为三种模式共用的主要展示区域：

```text
等候区
快充区 F1/F2
慢充区 T1/T2/T3
车辆卡片
队列
故障状态
账单摘要
```

差异只体现在状态标识：

| 状态来源 | 展示方式 |
|---|---|
| 真实已提交 | 实线卡片、正常颜色 |
| 课程回放 | 带“课程”来源说明 |
| 预测事件 | 虚线、半透明、`预测` 标签 |
| 历史快照 | 显示“历史时间”提示 |

### 7.5 右侧检查器

右侧不再叫“验收”，而叫 `事件详情` 或 `运行检查`。

内容随模式变化：

| 模式 | 检查器内容 |
|---|---|
| 实时 | 当前车辆、当前桩、最新操作、系统提示 |
| 回放 | 当前事件、前后变化、触发规则、课程核对 |
| 预测 | 预测假设、风险提示、预计等待时间、分支指标 |

课程核对只在来源包含 `COURSE_SEQUENCE` 时出现，并默认折叠到 `课程结果` 区域。

## 8. API 设计

第一期不必一次性重写所有接口。建议保留现有接口，同时新增一组时间系统接口。

### 8.1 事件来源

```text
GET /api/time/sources
POST /api/time/sources/course-sample/load
POST /api/time/sources/imported-plan
```

用途：

1. 查询可用事件来源。
2. 加载课程样例。
3. 导入未来订单或外部计划。

现有 `/api/scenarios/course-sample/run` 可继续保留，并作为课程来源适配器的兼容接口。

### 8.2 时间会话

```text
POST /api/time/sessions
GET /api/time/sessions/{sessionId}
PATCH /api/time/sessions/{sessionId}
DELETE /api/time/sessions/{sessionId}
```

创建会话请求示例：

```json
{
  "mode": "REPLAY",
  "sourceIds": ["course-sample"],
  "windowStart": "2026-06-14T06:00:00",
  "windowEnd": "2026-06-14T09:30:00",
  "clockRate": 60
}
```

### 8.3 快照投影

```text
GET /api/time/sessions/{sessionId}/snapshot?time=2026-06-14T08:25:00
GET /api/station/snapshot
```

`/api/station/snapshot` 保持当前实时快照接口定位；新的 session snapshot 用于回放和预测。

第一期可继续让回放接口一次性返回完整快照数组：

```text
POST /api/time/sessions/{sessionId}/precompute
```

这只是性能和播放稳定性的选择，不改变统一模型。

### 8.4 实时预制操作

```text
POST /api/events/preset
```

请求示例：

```json
{
  "presetType": "FAST_REQUEST",
  "eventTimePolicy": "NOW",
  "payload": {
    "requestKwh": "30.00"
  }
}
```

该接口用于课堂演示中的“点一下生成当前时间真实事件”。它应复用正常业务命令，不应走课程回放逻辑。

### 8.5 预测分支

```text
POST /api/forecast/branches
GET /api/forecast/branches/{branchId}
POST /api/forecast/branches/{branchId}/run
POST /api/forecast/branches/{branchId}/discard
```

第一期不做 `promote`。等预测展示稳定后，再考虑“应用为预约计划”。

## 9. 与现有代码的衔接

当前仓库已有：

1. `AcceptanceScenarioService`：课程序列和验收结果。
2. `ScenarioController`：课程回放 API。
3. `StationSnapshotService`：实时站点快照。
4. `SimulationSandbox.vue` 和 simulation 组件：当前调度沙盘页面。
5. `simulationPlayback.js`：前端播放状态机。

下一步不应推翻这些成果，而应在其上收敛：

### 9.1 前端先统一体验

把 `SimulationSandbox.vue` 演进为 `StationRuntimeView.vue` 或保持文件名但改内部结构：

```text
StationRuntimeView
  TimeModeTabs
  RuntimeClockPanel
  EventSourcePanel
  StationMap
  EventTimeline
  RuntimeInspector
```

第一期可以先只接入 `LIVE` 和 `REPLAY`，预测入口可以做成只读入口，展示预测分支的概念、状态和后续操作位置，但文案应说明它是“预测分支”，不是另一套演示页。

### 9.2 后端先统一 DTO 语义

当前 `ScenarioDtos.ReplayBundle` 可以逐步补充：

```text
session
sourceSummary
events.sourceType
events.commitState
snapshots.sessionMode
snapshots.provenance
```

不用立刻把数据库改成事件溯源，但 DTO 上要先表达“这是某个时间会话的投影结果”。

### 9.3 实时业务先接入快照

车主端和运营管理端在执行成功后，应统一刷新或接收 `StationSnapshot`。这样可以减少“清空数据后某个端还显示旧数据”的割裂感。

现有接口可以逐步调整为：

```text
业务操作结果 + 最新站点快照摘要
```

### 9.4 课程核对保留但降级

课程核对不再作为顶级导航、主标题或主按钮出现。

它的位置：

```text
站点运行 -> 回放 -> 事件来源：课程事件序列 -> 运行检查 -> 课程结果
```

这样老师验收时仍然能看到关键行通过，同学演示时也不会显得像只是在跑测试表。

## 10. 第一期实现路线

### 阶段一：站点运行台信息架构

目标：

1. 将当前沙盘页面升级为 `站点运行台` 语义。
2. 增加 `实时 / 回放 / 预测` 模式切换。
3. 回放模式继续使用现有课程序列。
4. 实时模式显示当前 `/api/station/snapshot`。
5. 预测模式先展示设计入口，不生成复杂结果。

验收标准：

1. 用户能看出三个模式属于同一个页面。
2. 页面文案不再把课程样例当作唯一主线。
3. 课程结果仍可展开核对。

### 阶段二：事件来源和时间会话 DTO

目标：

1. 给课程回放包补充 `sourceType`、`sessionMode`、`sourceSummary`。
2. 前端基于这些字段展示来源和状态。
3. 给实时快照补充当前时间、来源说明和基础指标。

验收标准：

1. 课程回放和实时快照在页面上使用同一套展示组件。
2. 同一辆车、同一个桩在不同模式下字段含义一致。

### 阶段三：实时预制操作

目标：

1. 增加“生成快充请求”“生成慢充请求”“模拟故障”等预制操作。
2. 预制操作走真实业务接口并刷新站点快照。
3. 预制操作在事件详情中显示为 `LIVE_PRESET`。

验收标准：

1. 演示者可以不手填表单也能制造实时事件。
2. 车主端和运营管理端能看到同一真实状态。

### 阶段四：预测分支最小闭环

目标：

1. 支持创建一个预测分支。
2. 支持添加未来请求事件。
3. 支持生成预测快照。
4. 预测车辆和预测事件用明确视觉样式区分。
5. 不支持应用到真实站点。

验收标准：

1. 用户能比较当前真实状态和未来预测状态。
2. 预测不会污染真实业务数据。

## 11. 可视化库取舍

### 11.1 Open MCT

Open MCT 适合复杂任务控制、遥测、时间序列和插件化工作台。它的理念值得借鉴：

1. 全局时间上下文。
2. 多数据源。
3. 时间序列对象。
4. 可组合视图。

但它不适合作为本项目第一期直接依赖：

1. 应用体量和概念复杂度偏大。
2. 会改变当前 Vue + Element Plus 的应用外壳。
3. 课程项目的核心风险在业务正确性，不在遥测平台能力。

结论：借鉴架构，不直接引入。

参考：[Open MCT](https://github.com/nasa/openmct)

### 11.2 Cesium

Cesium 的 `Clock` 和 `Timeline` 模型对本项目有参考价值：

1. 有当前时间。
2. 有开始和结束时间。
3. 有倍率。
4. 有播放和暂停。
5. 能表达时间范围内的状态。

但 Cesium 主要服务三维地理空间场景。当前系统不需要地球、地图或 3D 场景，直接引入会增加构建体积和概念负担。

结论：借鉴 `Clock` / `Timeline` 思想，不引入 Cesium。

参考：[Cesium Clock](https://cesium.com/learn/cesiumjs/ref-doc/Clock.html)、[Cesium Timeline](https://cesium.com/learn/cesiumjs/ref-doc/Timeline.html)

### 11.3 vis-timeline

`vis-timeline` 更接近本项目的事件轴需求，适合多泳道事件、缩放、拖动和点击。

但第一期不建议马上引入：

1. 现有事件数量有限。
2. 当前自研时间轴已经可控。
3. 引入后需要重新适配样式、测试和移动端。

结论：作为第二期候选。若课程序列、预测事件和实时历史明显增多，再评估引入。

参考：[vis-timeline](https://visjs.github.io/vis-timeline/docs/timeline/)

### 11.4 当前推荐

第一期使用 Vue + Element Plus + 自研轻量时间控制组件。

理由：

1. 与现有代码一致。
2. 最容易保证老师样例通过。
3. 对课堂演示足够。
4. 后续可以把时间轴局部替换为第三方库，而不推翻整体架构。

## 12. 测试策略

### 12.1 后端测试

必须覆盖：

1. 课程序列仍然通过现有核对项。
2. 回放包包含来源和会话信息。
3. 实时快照和回放快照共享关键字段语义。
4. 预制操作走真实业务流程，不调用课程回放服务。
5. 预测分支不写入真实业务数据。
6. 同一输入事件在同一规则下投影结果稳定。
7. 未来事件按 `eventTime + sequence` 稳定排序。

### 12.2 前端单元测试

必须覆盖：

1. `LIVE` 模式不显示倍率控制。
2. `REPLAY` 模式显示播放、暂停、单步、倍率和跳转。
3. `FORECAST` 模式显示预测来源和分支状态。
4. 切换模式后站点地图使用同一快照结构。
5. 课程来源存在时显示课程结果入口。
6. 非课程来源不显示课程核对为主内容。
7. 预测车辆有独立视觉标识。

### 12.3 浏览器验证

必须手动验证：

1. 桌面视口下时间控制区、站点地图、时间轴和检查器不重叠。
2. 移动端能操作模式切换和主按钮。
3. 实时模式刷新后车主端和运营管理端状态一致。
4. 回放模式播放、暂停、倍速、跳转不影响实时数据。
5. 预测模式生成未来事件后，退出预测不改变实时快照。
6. 课程结果仍然可以复制或展示给老师检查。

## 13. 风险与处理

| 风险 | 影响 | 处理方式 |
|---|---|---|
| 统一模型设计过大 | 拖慢课程交付 | 分阶段做，第一期只统一页面语义和 DTO |
| 实时和回放强行合并 | 破坏真实业务语义 | 用时间会话区分观察方式，不让实时站点倍速运行 |
| 预测和课程混淆 | 用户不知道哪些是真实状态 | 使用 `commitState` 和视觉标识区分 |
| 第三方库过重 | 构建复杂、样式割裂 | 第一阶段自研，库作为后续局部替换 |
| 课程核对被弱化 | 老师验收不方便 | 保留课程结果，但放在运行检查里 |
| 事件日志不完整 | 无法完全历史回放 | 第一阶段允许课程和预测预计算，实时回放后续再做 |

## 14. 设计决策

本文明确采用以下决策：

1. 系统的统一抽象是 `事件来源 + 时间会话 + 状态投影`。
2. 页面主入口应从单纯 `调度沙盘` 演进为 `站点运行` 或 `站点运行台`。
3. 实时、回放、预测是同一页面中的三种模式。
4. 课程事件序列是 `COURSE_SEQUENCE` 来源。
5. 课程结果核对是检查器，不是独立业务。
6. 实时模式不提供倍速控制。
7. 回放和预测可以提供播放、暂停、倍率、跳转。
8. 未来订单可以是真实业务的一部分，不应被简单归类为演示数据。
9. 预测事件默认是 `PROVISIONAL`，不会自动写入真实站点。
10. 第一期不直接引入 Open MCT、Cesium 或 vis-timeline。

## 15. 最小成功标准

下一轮迭代完成后，系统应满足：

1. 用户打开首页能看到一个统一的站点运行台，而不是孤立的测试页。
2. 用户能在同一页面理解实时、回放、预测的关系。
3. 课程样例仍能播放、单步、倍速、跳转、核对。
4. 实时模式可以显示当前站点，并能通过预制操作制造真实事件。
5. 预测模式至少在界面和数据结构上有清晰位置，不再需要另起一个割裂页面。
6. 车主自助和运营管理仍然保持可用。
7. 文档能说明为什么本系统没有直接引入 Open MCT 或 Cesium。

## 16. 后续实现建议

建议下一步只做第一期闭环：

1. 页面层把 `调度沙盘` 整理成 `站点运行台`。
2. 增加 `实时 / 回放 / 预测` 模式。
3. 回放模式复用当前课程沙盘。
4. 实时模式读取当前站点快照。
5. 预测模式先形成空分支或样例分支入口。
6. 调整文案，让“课程核对”只在回放检查器里出现。

这样可以让演示和正常使用在体验上先统一起来，同时不急着重写后端核心。等第一期稳定后，再做事件来源 API、时间会话 API 和预测分支的真实业务闭环。
